# 架构设计

## 问题

企业文档管理面临一个普遍的"M 对 N"问题：有 M 种文档来源（钉钉知识库、腾讯文档、本地文件...），N 种目标系统（Hindsight Memory、本地文件系统、S3...），每对来源-目标组合都需要不同的处理逻辑。传统做法是为每个组合写一个脚本，导致脚本数量暴增，维护成本高。

docupipe 的设计目标：用一套统一的框架，通过可配置的处理步骤，连接任意文档来源和目标系统。

## 方法

### 插件式 Pipeline 架构

```
Source ──→ [Step × N] ──→ Destination
                ↑
          Converter (内部分配)
```

四种组件类型各有明确的职责边界：

**Source** 回答"文档在哪里"。它知道如何列举文档列表和获取单个文档内容，但不关心内容如何处理或存到哪里。

**Destination** 回答"文档存到哪里"。它接收一个完整的 Bundle（文件+上下文），负责持久化。

**Step** 回答"文档要做什么处理"。它是纯函数式的：输入一个 Bundle，输出一个 Bundle。不关心数据从哪来到哪去。

**Converter** 回答"文件怎么转 Markdown"。它是最底层的格式转换引擎，被 Step 调度。

这种边界划分使得每个组件可以独立测试和替换。需要一个新的文档来源？写一个 Source。需要一个新的目标系统？写一个 Destination。不需要改动其他任何代码。

### Bundle：文档的通用表示

Bundle 是所有组件间传递的唯一数据容器：

```
Bundle
  ├── files: list[FileItem]    # 主文件 + 附件
  │     ├── main (role="main")        # 主文档（Markdown 或原始格式）
  │     ├── images (role="image")     # 内联图片
  │     └── attachments               # 其他附件
  └── context: dict            # 元数据键值对
```

使用 Bundle 而不是简单字符串的原因是：文档处理过程中会产生多种副产品（提取的图片、转换后的格式、AI 生成的元数据），它们需要和主文档一起传递。同一个 Bundle 在 pipeline 中流动，每个 step 可以添加、修改文件或更新 context。

### 装饰器注册模式

所有组件使用相同的注册模式：

1. 模块级字典（`SOURCES`、`DESTINATIONS`、`STEPS`、`CONVERTERS`）
2. `@register_xxx("name")` 装饰器将类注册到字典
3. `__init__.py` 中的 import 语句触发注册

选择装饰器而不是自动发现或配置文件声明的理由是：（1）显式 import 让依赖关系清晰可循；（2）装饰器可以在注册时做参数校验；（3）不需要额外的插件发现机制。

### 状态驱动的运行模式

状态文件是 pipeline 的"记忆"。四种运行模式围绕状态文件设计：

```
full:        list → 逐个处理 → 全部标记 done
full+resume: 不 list → 从状态找 pending → 继续处理
incremental: list → 只处理新增 → 标记 done
mirror:      list → 检测 mtime/hash 变化 → 清理消失的文档
```

状态文件的存在使得 pipeline 可以安全中断和恢复。状态文件操作是幂等的：重复执行同一个 pipeline 不会重复处理已完成的文档。

## 权衡

### 配置文件 vs 命令行

选择 YAML 配置文件作为主要配置方式，而不是纯 CLI 参数。这让复杂配置（嵌套的参数、多步骤 pipeline）更具可读性，也便于版本管理。代价是简单场景需要写文件而不能一行命令搞定。

### 同步 vs 异步

pipeline 目前是同步逐文档处理。这简化了错误处理和状态管理——如果一个文档处理失败，状态不会写入，重试时可以继续。但大量小文件的场景下单线程是瓶颈。异步支持可以放在未来考虑。

### 装饰器 vs 自动发现

装饰器注册要求每个新组件在 `__init__.py` 中有显式 import。这让 IDE 可以追踪依赖关系，但也意味着添加组件需要修改两个文件（组件本身 + `__init__.py`）。

### Context 字典 vs 类型安全

Bundle.context 是一个非类型化的字典，source 和 step 通过字段注册表约定来通信。这提供了最大灵活性（任何组件可以在任何阶段写入任何信息），但缺乏编译期类型检查。字段注册表（models.py 顶部）是缓解这一问题的文档化约束。

## 备选方案

在设计和演进过程中，考虑过以下替代方案：

**纯脚本方案**：为每个来源-目标组合写独立 Python 脚本。被否决因为 N×M 组合数增长时维护成本过高。

**任务队列系统**（Celery 等）：适合分布式场景但对简单的文档传输来说太重。当前项目使用场景主要在同机或内网环境。

**数据库作为状态**：JSON 文件作为状态存储而不是数据库。好处是零外部依赖，状态文件可读可审计。文档数量很大时可以考虑迁移到数据库方案。
